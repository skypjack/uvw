24 # Introduction
25 
26 `uvw` is a header-only, event based, tiny and easy to use *libuv* wrapper in
27 modern C++.<br/>
28 The basic idea is to hide completely the *C-ish* interface of *libuv* behind a
29 graceful C++ API. Currently, no `uv_*_t` data structure is actually exposed by
30 the library.<br/>
31 Note that `uvw` stays true to the API of *libuv* and it doesn't add anything to
32 its interface. For the same reasons, users of the library must follow the same
33 rules who are used to follow with *libuv*.<br/>
34 As an example, a *handle* should be initialized before any other operation and
35 closed once it is no longer in use.
36 
37 ## Code Example
38 
39 ```cpp
40 #include <uvw.hpp>
41 #include <memory>
42 
43 void listen(uvw::Loop &loop) {
44  std::shared_ptr<uvw::TCPHandle> tcp = loop.resource<uvw::TCPHandle>();
45 
46  tcp->once<uvw::ListenEvent>([](const uvw::ListenEvent &, uvw::TCPHandle &srv) {
47  std::shared_ptr<uvw::TCPHandle> client = srv.loop().resource<uvw::TCPHandle>();
48 
49  client->on<uvw::CloseEvent>([ptr = srv.shared_from_this()](const uvw::CloseEvent &, uvw::TCPHandle &) { ptr->close(); });
50  client->on<uvw::EndEvent>([](const uvw::EndEvent &, uvw::TCPHandle &client) { client.close(); });
51 
52  srv.accept(*client);
53  client->read();
54  });
55 
56  tcp->bind("127.0.0.1", 4242);
57  tcp->listen();
58 }
59 
60 void conn(uvw::Loop &loop) {
61  auto tcp = loop.resource<uvw::TCPHandle>();
62 
63  tcp->on<uvw::ErrorEvent>([](const uvw::ErrorEvent &, uvw::TCPHandle &) { /* handle errors */ });
64 
65  tcp->once<uvw::ConnectEvent>([](const uvw::ConnectEvent &, uvw::TCPHandle &tcp) {
66  auto dataWrite = std::unique_ptr<char[]>(new char[2]{ 'b', 'c' });
67  tcp.write(std::move(dataWrite), 2);
68  tcp.close();
69  });
70 
71  tcp->connect(std::string{"127.0.0.1"}, 4242);
72 }
73 
74 int main() {
75  auto loop = uvw::Loop::getDefault();
76  listen(*loop);
77  conn(*loop);
78  loop->run();
79 }
80 ```

82 ## Motivation
83 
84 The main reason for which `uvw` has been written is the fact that it does not
85 exist a valid *libuv* wrapper in C++. That's all.

87 # Build Instructions
88 
89 ## Requirements
90 
91 To be able to use `uvw`, users must provide the following system-wide tools:
92 
93 * A full-featured compiler that supports at least C++14.
94 * `libuv` (which version depends on the tag of `uvw` in use).
95 
96 The requirements below are mandatory to compile the tests and to extract the
97 documentation:
98 
99 * CMake version 3.2 or later.
100 * Doxygen version 1.8 or later.
101 
102 Note that `libuv` is part of the dependencies of the project and it will be
103 cloned by `cmake` (see below for further details).<br/>
104 Because of that, users have not to install it to compile and execute the tests.
105 
106 ## Library
107 
108 `uvw` is a header-only library.<br/>
109 This means that including the `uvw.hpp` header or one of the other `uvw/*.hpp`
110 headers is enough to use it.<br/>
111 It's a matter of adding the following line at the top of a file:
112 
113 ```cpp
114 #include <uvw.hpp>
115 ```
116 
117 Then pass the proper `-I` argument to the compiler to add the `src` directory to
118 the include paths.<br/>
119 Note that users are demanded to correctly setup include directories and
120 libraries search paths for *libuv*.
121 
122 ## Versioning
123 
124 Starting with tag _v1.12.0_ of `libuv`, `uvw` follows the
125 [semantic versioning](http://semver.org/) scheme.<br/>
126 The problem is that any version of `uvw` also requires to track explicitly the
127 version of `libuv` to which it is bound.<br/>
128 Because of that, the latter wil be appended to the version of `uvw`. As an
129 example:
130 
131  vU.V.W_libuv-vX.Y
132 
133 In particular, the following applies:
134 
135 * _U.V.W_ are major, minor and patch versions of `uvw`.
136 * _X.Y_ is the version of `libuv` to which to refer (where any patch version is
137  valid).
138 
139 In other terms, tags will look like this from now on:
140 
141  v1.0.0_libuv-v1.12
142 
143 Branch `master` of `uvw` will be a work in progress branch that follows branch
144 _v1.x_ of `libuv` (at least as long as it remains their _master_ branch).<br/>

189 # Crash Course
190 
191 ## Vademecum
192 
193 There is only one rule when using `uvw`: always initialize the resources and
194 terminate them.
195 
196 Resources belong mainly to two families: _handles_ and _requests_.<br/>
197 Handles represent long-lived objects capable of performing certain operations
198 while active.<br/>
199 Requests represent (typically) short-lived operations performed either over a
200 handle or standalone.
201 
202 The following sections will explain in short what it means to initialize and
203 terminate these kinds of resources.<br/>
204 For more details, please refer to the
205 [online documentation](https://skypjack.github.io/uvw/).
206 
207 ## Handles
208 
209 Initialization is usually performed under the hood and can be even passed over,
210 as far as handles are created using the `Loop::resource` member function.<br/>
211 On the other side, handles keep themselves alive until one explicitly closes
212 them. Because of that, memory usage will grow up if users simply forget about a
213 handle.<br/>
214 Therefore the rule quickly becomes *always close your handles*. It's as simple
215 as calling the `close` member function on them.
216 
217 ## Requests
218 
219 Usually initializing a request object is not required. Anyway, the recommended
220 way to create a request is still through the `Loop::resource` member
221 function.<br/>
222 Requests will keep themselves alive as long as they are bound to unfinished
223 underlying activities. This means that users have not to discard explicitly a
224 request.<br/>
225 Therefore the rule quickly becomes *feel free to make a request and forget about
226 it*. It's as simple as calling a member function on them.
227 
228 ## The Loop and the Resource
229 
230 The first thing to do to use `uvw` is to create a loop. In case the default one
231 is enough, it's easy as doing this:
232 
233 ```cpp
234 auto loop = uvw::Loop::getDefault();
235 ```
236 
237 Note that loop objects don't require to be closed explicitly, even if they offer
238 the `close` member function in case an user wants to do that.<br/>
239 Loops can be started using the `run` member function. The two calls below are
240 equivalent:
241 
242 ```cpp
243 loop->run();
244 loop->run<uvw::Loop::Mode::DEFAULT>();
245 ```
246 
247 Available modes are: `DEFAULT`, `ONCE`, `NOWAIT`. Please refer to the
248 documentation of *libuv* for further details.
249 
250 In order to create a resource and to bind it to the given loop, just do the
251 following:
252 
253 ```cpp
254 auto tcp = loop.resource<uvw::TCPHandle>();
255 ```
256 
257 The line above will create and initialize a tcp handle, then a shared pointer to
258 that resource will be returned.<br/>
259 Users should check if pointers have been correctly initialized: in case of
260 errors, they won't be.<br/>
261 Another way to create a resource is:
262 
263 ```cpp
264 auto tcp = TCPHandle::create(loop);
265 tcp->init();
266 ```
267 
268 Pretty annoying indeed. Using a loop is the recommended approach.
269 
270 The resources also accept arbitrary user-data that won't be touched in any
271 case.<br/>
272 Users can set and get them through the `data` member function as it follows:
273 
274 ```cpp
275 resource->data(std::make_shared<int>(42));
276 std::shared_ptr<void> data = resource->data();
277 ```
278 
279 Resources expect a `std::shared_pointer<void>` and return it, therefore any kind
280 of data is welcome.<br/>
281 Users can explicitly specify a type other than `void` when calling the `data`
282 member function:
283 
284 ```cpp
285 std::shared_ptr<int> data = resource->data<int>();
286 ```
287 
288 Remember from the previous section that a handle will keep itself alive until
289 one invokes the `close` member function on it.<br/>
290 To know what are the handles that are still alive and bound to a given loop,
291 just do the following:
292 
293 ```cpp
294 loop.walk([](uvw::BaseHandle &){ /* application code here */ });
295 ```
296 
297 `BaseHandle` exposes a few methods and cannot be promoted to the original type
298 of the handle (even though `type` and `category` member functions fill the gap
299 somehow).<br/>
300 Anyway, it can be used to close the handle that originated from it. As an
301 example, all the pending handles can be closed easily as it follows:
302 
303 ```cpp
304 loop.walk([](uvw::BaseHandle &h){ h.close(); });
305 ```
306 
307 No need to keep track of them.
308 
309 To know what are the available resources' types, please refer the API reference.
310 
311 ## The event-based approach
312 
313 For `uvw` offers an event-based approach, resources are small event emitters
314 to which listeners can be attached.<br/>
315 Attaching a listener to a resource is the recommended way to be notified about
316 changes.<br/>
317 Listeners must be callable objects of type `void(EventType &, ResourceType &)`,
318 where:
319 
320 * `EventType` is the type of the event for which they have been designed.
321 * `ResourceType` is the type of the resource that has originated the event.
322 
323 It means that the following function types are all valid:
324 
325 * `void(EventType &, ResourceType &)`
326 * `void(const EventType &, ResourceType &)`
327 * `void(EventType &, const ResourceType &)`
328 * `void(const EventType &, const ResourceType &)`
329 
330 Once more, please note that there is no need to keep around references to the
331 resources: they will pass themselves as an argument whenever an event is
332 published.
333 
334 There exist two methods to attach an event to a resource:
335 
336 * `resource.once<EventType>(listener)`: the listener will be automatically
337  removed after the first event of the given type.
338 * `resource.on<EventType>(listener)`: to be used for long-running listeners.
339 
340 Both of them return an object of type `ResourceType::Connection` (as an example,
341 `TCPHandle::Connection`).<br/>
342 A connection object can be used later as an argument to the `erase` member
343 function of the resource to remove the listener.<br/>
344 There exists also the `clear` member function to drop all the listeners at once.
345 
346 Almost all the resources use to emit `ErrorEvent` events in case of errors.<br/>
347 All the other events are specific for the given resource and documented in the
348 API reference.
349 
350 The code below shows how to create a simple tcp server using `uvw`:
351 
352 ```cpp
353 auto loop = uvw::Loop::getDefault();
354 auto tcp = loop.resource<uvw::TCPHandle>();
355 
356 tcp->on<uvw::ErrorEvent>([](const uvw::ErrorEvent &, uvw::TCPHandle &) { /* something went wrong */ });
357 
358 tcp->on<uvw::ListenEvent>([](const uvw::ListenEvent &, uvw::TCPHandle &srv) {
359  std::shared_ptr<uvw::TCPHandle> client = srv.loop().resource<uvw::TCPHandle>();
360  client->once<uvw::EndEvent>([](const uvw::EndEvent &, uvw::TCPHandle &client) { client.close(); });
361  client->on<uvw::DataEvent>([](const uvw::DataEvent &, uvw::TCPHandle &) { /* data received */ });
362  srv.accept(*client);
363  client->read();
364 });
365 
366 tcp->bind("127.0.0.1", 4242);
367 tcp->listen();
368 ```
369 
370 Note also that `uvw::TCPHandle` already supports _IPv6_ out-of-the-box. The
371 statement above is equivalent to `tcp->bind<uvw::IPv4>("127.0.0.1", 4242)`.<br/>
372 It's suffice to explicitly specify `uvw::IPv6` as the underlying protocol to use
373 it.
374 
375 The API reference is the recommended documentation for further details about
376 resources and their methods.
377 
378 ## Going raw
379 
380 In case users need to use functionalities not wrapped yet by `uvw` or if they
381 want to get the underlying data structures as defined by `libuv` for some other
382 reasons, almost all the classes in `uvw` give direct access to them.<br/>
383 Please, note that this functions should not be used directly unless users know
384 exactly what they are doing and what are the risks. Going raw is dangerous,
385 mainly because the lifetime management of a loop, a handle or a request is
386 completely in charge to the library and working around it could quickly break
387 things.
388 
389 That being said, _going raw_ is a matter of using the `raw` member functions:
390 
391 ```cpp
392 auto loop = uvw::Loop::getDefault();
393 auto tcp = loop.resource<uvw::TCPHandle>();
394 
395 uv_loop_t *raw = loop->raw();
396 uv_tcp_t *handle = tcp->raw();
397 ```
398 
399 Go the raw way at your own risk, but do not expect any support in case of bugs.
